# CRM 검색어 확장 하이브리드 검색기 설계

## 목표

사용자 질문에서 원문과 확장 검색어를 만들고, `crm_entities`의 BM25 키워드 검색과 pgvector 의미 검색 결과를 RRF로 결합하는 하이브리드 검색기를 제공한다. CLI와 Gradio에서 동일한 검색 서비스를 사용한다.

## 접근 방식

세 가지 결합 방식 중 RRF를 선택한다.

- RRF 순위 결합: BM25와 코사인 유사도의 점수 범위를 별도로 보정할 필요가 없고 안정적이다.
- 가중 점수 합산: 가중치와 점수 정규화를 데이터별로 튜닝해야 한다.
- LLM 재정렬: 품질 향상 가능성은 있지만 비용, 지연, 결과 변동성이 커진다.

## 검색 파이프라인

1. 사용자가 한국어 질문을 입력한다.
2. 검색어 확장기가 원문 의도를 유지하는 보조 검색어를 최대 3개 생성한다.
3. 원문과 각 보조 검색어를 BM25와 pgvector 검색에 전달한다.
4. 각 검색기의 상위 후보 순위를 RRF로 합산한다.
5. 엔터티 키별로 중복을 제거하고 최종 상위 결과를 반환한다.

검색어 확장은 OpenAI 구조화 출력을 사용한다. API 키가 없거나 호출이 실패하거나 잘못된 결과를 반환하면 원문 검색어만 사용한다. 공백 또는 원문과 동일한 확장어는 제거한다.

## BM25 검색

`crm_entities` 97개를 메모리에 읽고 `rank-bm25`의 BM25Okapi 인덱스를 만든다. 한국어 형태소 분석기 의존성 없이 다음 토큰을 함께 사용한다.

- 공백 기준 소문자 단어 토큰
- 공백을 제거한 문자열의 문자 바이그램

엔터티의 `content`, `entity_name`, `entity_type`을 검색 문서로 사용한다. 빈 질문은 거부하고 점수가 0보다 큰 결과만 반환한다.

## 벡터 검색

기존 `text-embedding-3-small` 1536차원 쿼리 임베딩과 pgvector 코사인 거리를 사용한다. 검색 결과에는 엔터티 키, 타입, 이름, 내용, 출처 수와 코사인 유사도를 포함한다.

## RRF 결합

각 `(검색어, 검색기)` 결과의 순위 `rank`에 대해 `1 / (60 + rank)`를 더한다. 동일 엔터티는 하나로 합치며 다음 진단 정보를 유지한다.

- BM25 최고 점수
- 벡터 최고 유사도
- RRF 통합 점수
- 매칭된 검색어 목록
- 검색기에 등장한 순위 근거

동점은 `entity_type`, `entity_name` 순으로 정렬해 결정론적으로 만든다.

## 인터페이스

- `crm/hybrid_search.py`: 토큰화, BM25, 검색어 확장, RRF 결합 및 CLI
- CLI: `python -m crm.hybrid_search "휴면 고객 할인" --limit 10`
- `app.py`: “하이브리드 검색” 탭에서 질문, 확장 검색어, 통합 결과 표를 표시

## 오류 및 안전성

- 빈 질문과 1 미만 limit을 거부한다.
- OpenAI 장애는 원문-only 검색으로 fallback한다.
- DB 연결 오류는 자격 증명 없이 사용자 안내로 변환한다.
- 검색은 읽기 전용이며 `crm_events`와 `crm_entities`를 수정하지 않는다.
- `app_old.py`, `etf_database.db`는 수정하지 않는다.

## 검증

- 한국어 단어·문자 바이그램 토큰화 단위 테스트
- BM25가 정확한 고유명사를 상위에 반환하는 테스트
- 검색어 확장의 중복·공백·장애 fallback 테스트
- RRF 중복 제거, 순위 합산, 결정론적 동점 테스트
- 가짜 embedder와 테스트 DB 결과를 사용한 하이브리드 서비스 테스트
- 실제 DB에서 대표 질문 검색 결과 확인
- Gradio 앱 import 및 탭 생성 테스트
- 전체 테스트와 Python 컴파일 성공
- 기존 DB 행 수와 보호 원본 해시 유지
